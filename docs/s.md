## 2. 要求仕様書（顧客ヒアリング結果）

### 2.1 顧客の要望（ヒアリングメモ）

> ここに、顧客（講師）へのヒアリングで得た要望を箇条書きで記録してください。

* 商品一覧を確認できること
* 商品の登録・編集・削除ができること
* 購入時に在庫が自動で減算されること
* 金額投入と釣銭計算ができること
* 失敗入力でもプログラムが停止せず再入力を促すこと
* 取引や操作のログをCSV形式で記録すること

## 3. 非機能要件

| 区分 | 要件 |
| :--- | :--- |
| 可用性 | 入力異常やログ書き込み失敗が発生しても、プログラム全体は終了せずメニューに復帰できること。 |
| 性能 | 通常操作（メニュー表示、商品選択、購入確定）の応答は1秒以内を目標とする。 |
| 保守性 | 商品数上限（50件）や税計算方式などの定数を、1か所の定義変更で反映できる構造にする。 |
| ログ品質 | 全ログに日時、操作種別、結果（OK/NG）、セッションIDを含めること。 |
| 操作性 | 初見利用者でも、3ステップ以内（選択→数量→投入）で購入完了できること。 |

## 4. 制約条件・前提

| 項目 | 内容 |
| :--- | :--- |
| 商品種類数 | 同時に管理できる商品は最大50種類。 |
| 在庫数 | 各商品の在庫は0以上50以下。 |
| 入力方式 | キーボードによる数値入力を基本とし、無効入力時は再入力させる。 |
| データ保存 | 永続化はCSVファイルで行い、文字コードはUTF-8を使用する。 |
| 同時利用 | 複数利用者はセッション単位で順次処理し、同一在庫への更新競合を防ぐ。 |

## 5. 受け入れ基準

| ID | 受け入れ基準 |
| :--: | :--- |
| A01 | 商品登録時、51件目の追加は拒否され、エラーメッセージが表示される。 |
| A02 | 在庫を超える数量入力は確定されず、再入力を求める。 |
| A03 | 購入確定時に合計金額と釣銭が正しく表示され、在庫が減算される。 |
| A04 | 1取引ごとに販売ログCSVと操作ログCSVへ1行以上追記される。 |
| A05 | 数字以外入力や不正商品ID入力でもプログラムが停止せず、メニューに戻れる。 |
| A06 | エラーログには故障予知用としてエラー種別と発生時刻が記録される。 |

## 7. CSV保存形式

- 販売ログCSV: 取引ID,セッションID,日時,商品ID,商品名,単価,数量,小計,投入金額,釣銭,結果
- 操作ログCSV: ログID,セッションID,日時,操作種別,詳細,結果,エラーコード
- 故障予知CSV: 記録ID,日時,エラー種別,発生箇所,入力値,メッセージ,重大度

## 4. 詳細設計（アルゴリズム）

### 4.1 メインループ

> メニュー表示→入力受付→処理分岐→ループの流れを記述してください。
> フローチャートまたは疑似コード（箇条書き）で記述してください。

```
1. 起動時に商品マスタとログCSVを読み込む
2. メインメニューを表示する
3. ユーザーの入力を受け取る
4. 入力に応じて処理を分岐する
   - 1: 商品一覧表示
   - 2: 商品登録
   - 3: 商品情報編集
   - 4: 商品削除
   - 5: 購入処理
   - 6: ログ表示/CSV保存
   - 9: 終了
5. 各処理終了後、メインメニューに戻る
6. 9が入力されたら保存処理を行いプログラムを終了する
```

### 4.2 個別機能のアルゴリズム

#### 機能1: 商品一覧表示

```
1. 商品マスタを先頭から順に読み込む
2. 商品ID、商品名、価格、在庫数をテーブル形式で表示する
3. 一覧表示後、任意のキー入力でメインメニューに戻る
```

#### 機能2: 商品登録

```
1. 新規登録可能件数を確認する（最大50件）
2. 商品名を入力させる
3. 価格を入力させる
4. 在庫数を入力させる
5. 入力値を検証する
   - カウントが50件以上なら登録を拒否
   - 価格・在庫が負数でないか確認
6. 登録可能なら商品マスタに追加する
7. 操作ログに登録結果を追記する
```

#### 機能3: 商品情報編集

```
1. 編集対象の商品IDを入力させる
2. 指定IDの商品が存在するか確認する
3. 編集する項目を選択させる（商品名、価格、在庫数）
4. 新しい値を入力させる
5. 入力値を検証する
6. データを更新し、変更内容を保存候補として保持する
7. 操作ログに編集結果を追記する
```

#### 機能4: 商品削除

```
1. 削除対象の商品IDを入力させる
2. 指定IDの商品が存在するか確認する
3. 削除の確認を行う
4. ユーザーが確定したら商品をマスタから削除する
5. 操作ログに削除結果を追記する
```

#### 機能5: 購入処理（商品選択〜購入確定）

```
1. 商品一覧を表示する
2. 購入する商品IDを入力させる
3. 商品IDの存在と在庫数を確認する
4. 購入数量を入力させる
5. 在庫数を超える数量が入力された場合は再入力を促す
6. 合計金額を計算し表示する
7. 投入金額を入力させる
8. 投入金額が合計金額未満の場合は差額を表示して再入力を促す
9. 釣銭を計算する
10. 在庫を減算する
11. 販売ログCSVに取引情報を追記する
12. 操作ログCSVに購入処理結果を追記する
13. 購入完了メッセージを表示しメインメニューに戻る
```

#### 機能6: ログ保存と異常ログ記録

```
1. 各操作・取引の後にログをCSVに追記する
2. CSV書き込みに失敗した場合はエラーメッセージを表示し、再試行またはメニューへ復帰する
3. 想定外のエラーや不正入力は故障予知CSVに記録する
```

### 4.3 エラーハンドリング（想定される異常系）

> ユーザーが「想定外の操作」をした場合にどうするか検討してください。

| 異常系シナリオ | 対処方針 |
| :--- | :--- |
| 数字以外の文字が入力された場合 | 入力エラーとしてメッセージを表示し、再入力を促す |
| 存在しない商品IDが入力された場合 | エラーメッセージを表示し、商品ID入力に戻す |
| 在庫を超える数量が入力された場合 | 再入力を促し、指定数量が在庫以下になるまで繰り返す |
| 投入金額が合計より少ない場合 | 不足分を表示し、再投入を促す |
| CSVファイルが書き込み不可の場合 | エラーメッセージを表示してメニューへ復帰し、ログ保存は次回起動時に再試行可能とする |

## 5. 備考・メモ

> 設計中に気になった点や、明日の実装に向けた注意点があれば記録してください。

* 商品数上限は50件なので登録時に件数チェックを必ず入れる
* CSVはUTF-8で扱い、改行コードはOSに依存しない形式を意識する
* 入力検証は各画面で共通関数化し、エラー時には必ずメニューに戻れるようにする
* 操作ログと故障予知ログを分けて、通常の操作と異常系を区別する
* 画面表示は簡易テキストUIでも良いので、操作手順を3ステップ以内に収める
